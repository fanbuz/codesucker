# 大仓库性能基线

本文记录 issue #3 的可复现基线。它用于比较主线程同步实现与 worker 流水线，重点关注 Electron 主进程可响应性；数据不是不同机器间的绝对性能承诺。

## 运行方法

```bash
npm run perf
```

默认生成 5000 个 TypeScript 文件、每个 40 行，共 20 万行。可通过 `PERF_FILES` 与 `PERF_LINES` 调整规模。基准会校验同步与并发实现的文件顺序、截取结果和审计结果完全一致，并在结束后删除临时工程。

## 2026-07-22 基线

- 平台：darwin 27.0.0 arm64
- CPU：Apple M4 Pro，12 个逻辑核心
- Node.js：v20.19.4
- worker：4
- 数据集：5000 个文件，20 万行

| 阶段 | 同步耗时 | worker 耗时 | 同步最大事件循环延迟 | worker 最大事件循环延迟 | 同步峰值 RSS | worker 峰值 RSS |
|---|---:|---:|---:|---:|---:|---:|
| 扫描 | 235 ms | 825 ms | 225 ms | 5 ms | 235 MB | 306 MB |
| 清洗、截取与审计 | 1796 ms | 682 ms | 1786 ms | 11 ms | 401 MB | 435 MB |
| DOCX 渲染 | 143 ms | 109 ms | 70 ms | 2 ms | 469 MB | 504 MB |

## 结论与权衡

- worker 版本把三个重任务阶段的最大事件循环延迟控制在 11 ms 以内，达到主进程保持响应的首要目标。
- 清洗、截取与审计阶段耗时下降约 62%；DOCX 渲染耗时下降约 24%。
- 5000 个小文件的扫描总耗时增加，因为每文件任务的 worker 调度和结构化克隆成本高于直接读取；但主线程阻塞由 225 ms 降到 5 ms。当前实现优先保证界面可响应，后续可通过批量派发小文件减少调度开销。
- 并发会增加 worker 隔离堆和消息复制产生的内存开销，本次各阶段峰值增加约 34–71 MB。worker 数限制为 1–4，避免按 CPU 核数无限扩张。

每次调整并发数、任务粒度、扫描算法或渲染实现后，应在同一机器上重新运行此命令，并把新结果与本基线比较。
